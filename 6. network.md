# 네트워크

## 1. 웹 통신 개요

### ⭐ 웹 프로토콜이란

웹 프로토콜은 클라이언트와 서버가 데이터를 주고받기 위해 정한 통신 규칙입니다.
데이터 형식, 전송 방식, 요청·응답 구조 등을 정의하여
서로 다른 시스템 간 통신을 가능하게 합니다.

대표적인 웹 프로토콜로는 HTTP, HTTPS가 있습니다.

### ⭐ HTTP란

HTTP는 웹에서 클라이언트와 서버 간 데이터를 주고받기 위한 프로토콜입니다.
요청(Request)과 응답(Response) 구조로 동작하며, 무상태성(stateless)을 특징으로 합니다.

- 각 요청은 서로 독립적
- 서버는 이전 요청 상태를 저장하지 않음

→ 확장성과 단순성이 장점

## 2. 보안 통신 (HTTPS)

### ⭐ HTTP와 HTTPS 차이

- HTTP
  - 암호화 ❌
  - 포트 80
- HTTPS
  - SSL/TLS 기반 암호화
  - 포트 443
  - 중간자 공격 방지

→ HTTPS는 로그인, 결제 등 보안이 필요한 통신에 사용

### ⭐ HTTPS 동작 방식

HTTPS는 TLS Handshake 과정을 통해 암호화 통신을 설정합니다.

1. 클라이언트가 서버에 접속 요청
2. 서버는 인증서 전달
3. 클라이언트는 인증서 검증
4. 대칭 키 생성 및 공유
5. 이후 모든 데이터는 대칭 키로 암호화 통신

→ 인증 + 암호화를 동시에 보장

## 3. 전송 계층 (TCP / UDP)

### ⭐ TCP와 UDP 비교

- TCP
  - 연결 지향
  - 신뢰성 보장
  - 데이터 순서 보장
- UDP
  - 비연결
  - 빠르지만 신뢰성 낮음

→ TCP는 웹 통신,

→ UDP는 스트리밍, 실시간 게임 등에 적합

### ⭐ TCP가 신뢰성을 보장하는 방법

TCP는 다음 방식으로 신뢰성을 보장합니다.

- 패킷 순서 보장
- 오류 검출
- 손실 시 재전송
- 흐름 제어 및 혼잡 제어

→ 데이터 무결성과 정확한 전달 보장

### ⭐ TCP 3-Way Handshake

TCP 연결을 수립하는 과정입니다.

1. 클라이언트 → SYN
   (연결 요청)
2. 서버 → SYN + ACK
   (요청 수락 + 응답)
3. 클라이언트 → ACK
   (연결 확정)

→ 양측 연결 준비 완료

### ⭐ TCP 4-Way Handshake

TCP 연결을 종료하는 과정입니다.

1. FIN
   - 한쪽이 “더 이상 보낼 데이터가 없음”을 알림

2. ACK
   - 상대방이 FIN 수신 확인
   - 아직 보낼 데이터가 남아 있을 수 있음

3. FIN
   - 남은 데이터 전송 후
   - 상대방도 종료 의사 전달

4. ACK
   - 마지막 FIN 확인 후 연결 종료

→ 양방향 통신을 안전하게 종료하기 위한 과정

## 4. REST API & HTTP 메서드

### ⭐ REST API란

REST API는 자원을 URL로 표현하고,
HTTP 메서드(GET, POST, PUT, DELETE 등)로 행위를 표현하는 API 설계 방식입니다.

- 무상태성(stateless)
- 자원 중심 설계
- 클라이언트–서버 분리

### ⭐ HTTP GET과 POST 비교

- GET
  - 데이터 조회 목적
  - URL 쿼리스트링에 데이터 포함
  - 데이터 노출, 길이 제한 존재

- POST
  - 데이터 생성·전송 목적
  - 요청 본문(body)에 데이터 포함
  - 대용량·민감 데이터 전송에 적합

→ GET은 조회, POST는 제출·변경

### ⭐ HTTP PUT과 PATCH 비교

- PUT
  - 자원 전체 수정
  - 기존 자원을 완전히 대체
- PATCH
  - 자원 일부 수정
  - 변경된 필드만 전달

→ 전체 교체는 PUT, 부분 수정은 PATCH

### ⭐ HTTP Status Code란

서버가 요청 처리 결과를 숫자로 표현한 것입니다.

- 2xx: 성공
- 3xx: 리다이렉션
- 4xx: 클라이언트 오류
- 5xx: 서버 오류

→ 요청 결과를 명확하게 표현하기 위한 규칙
