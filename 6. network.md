## 네트워크

### ⭐ 웹 프로토콜이란

웹 프로토콜은 클라이언트와 서버가 데이터를 주고받기 위해 정한 통신 규칙입니다.
데이터 형식, 전송 방식, 요청·응답 구조 등을 정의해 서로 다른 시스템 간 통신을 가능하게 합니다.

### ⭐ HTTP란

HTTP는 웹에서 클라이언트와 서버 간 데이터를 주고받기 위한 프로토콜입니다.
요청(Request)과 응답(Response) 구조로 동작하며, **무상태성(stateless)**을 특징으로 합니다.

### ⭐ HTTP GET과 POST 비교

- GET
  - 데이터 조회 목적
  - URL 쿼리스트링에 데이터 포함
  - 데이터 노출, 길이 제한 존재
- POST
  - 데이터 생성·전송 목적
  - 요청 본문(body)에 데이터 포함
  - 대용량·민감 데이터 전송에 적합

→ GET은 조회, POST는 제출·변경에 사용

### ⭐ HTTP PUT과 PATCH 비교

- PUT
  - 자원 전체 수정
  - 기존 자원을 완전히 대체
- PATCH
  - 자원 일부 수정
  - 변경된 필드만 전달

→ 전체 교체는 PUT, 부분 수정은 PATCH

### ⭐ HTTP Status Code란

서버가 요청 처리 결과를 숫자로 표현한 것입니다.

- 2xx: 성공
- 3xx: 리다이렉션
- 4xx: 클라이언트 오류
- 5xx: 서버 오류

### ⭐ HTTP와 HTTPS 차이

- HTTP
  - 암호화 ❌
  - 포트 80
- HTTPS
  - SSL/TLS 기반 암호화
  - 포트 443
  - 중간자 공격 방지

→ HTTPS는 보안이 필요한 통신에 사용

### ⭐ HTTPS 동작 방식

클라이언트와 서버는 TLS Handshake를 통해 암호화 통신을 준비합니다.
서버의 인증서를 검증한 뒤, 대칭 키를 생성해 이후 데이터를 암호화하여 전송합니다.

### ⭐ TCP와 UDP 비교

- TCP
  - 연결 지향
  - 신뢰성 보장
  - 순서 보장
- UDP
  - 비연결
  - 빠르지만 신뢰성 낮음

→ TCP는 웹 통신, UDP는 스트리밍·게임에 적합

### ⭐ TCP가 신뢰성을 보장하는 방법

- 패킷 순서 보장
- 오류 검출
- 재전송
- 흐름 제어

→ 데이터 무결성과 정확한 전달 보장

### ⭐ TCP 3-Way Handshake

연결을 수립하는 과정입니다.

1. 클라이언트 → SYN
2. 서버 → SYN + ACK
3. 클라이언트 → ACK

→ 양측 연결 준비 완료

### ⭐ TCP 4-Way Handshake

연결을 종료하는 과정입니다.

1.  FIN

- 한쪽이 “나는 더 이상 보낼 데이터가 없다”는 의미로 FIN 전송

2. ACK

- 상대방이 FIN을 정상적으로 받았음을 응답
- 이 시점에서도 상대방은 아직 보낼 데이터가 남아 있을 수 있음

3. FIN

- 남은 데이터를 모두 전송한 뒤
- 상대방도 “나도 더 이상 보낼 데이터가 없다”는 의미로 FIN 전송

4. ACK

- 마지막 FIN을 확인하고 연결 종료

→ 양측 자원 정리 후 연결 종료
